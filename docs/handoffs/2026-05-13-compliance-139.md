## Gotchas

- **No Supabase branch testing.** An attempt to create a branch on `zwtnpyjifbdytlektxlc` came up `MIGRATIONS_FAILED`: Supabase's migration table on the parent only tracks 67 rows starting at `20260506174204`, but the actual schema (001‚Äì138) was applied out-of-band (via `supabase db push` or psql). A branch tries to replay only the tracked subset, which references base tables that don't exist in that subset, so the branch DB is empty and migration 139 has nothing to layer onto. The broken branch was deleted to stop the hourly meter. **If branch-based testing becomes a regular need, the parent's migration history needs reconciling first** ‚Äî separate workstream.
- **Status is computed, not stored.** Do not add a `status` column to `compliance_obligations`. If you need it in a query, call `deriveObligationStatus()` from `@soteria/core/compliance` server-side (as the `/api/compliance/obligations` GET route does) and attach the value to each row before returning.
- **Cadence math lives in core, not in SQL.** The complete route fetches `frequency` + `frequency_days` and calls `advanceNextDueDate()`. A trigger-based version was considered and rejected ‚Äî the rule belongs next to its tests, not buried in plpgsql.
- **AI output is never auto-persisted.** Both AI routes return a *proposal*; the admin clicks Accept to fire a PATCH. If you ever change this, also update `AGENTS.md` "humans confirm" guarantees.
